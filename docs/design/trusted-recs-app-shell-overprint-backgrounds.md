# Trusted Recs — UX Architecture & Overprint Background Supplement for Codex

## 0. Purpose

Use this supplement alongside:

- `PROJECT_BRIEF.md`
- `TRUSTED_RECS_FINAL_UX_DESIGN_GUIDE_FOR_CODEX_v2.md`
- the supplied approved design reference image

This document bundles the latest UX direction changes:

1. Move the core app away from long scrolling web pages.
2. Reframe key flows as app-like, near scroll-free wizard screens.
3. Use controlled internal scroll only where it is the correct interaction.
4. Replace autogenerated/random-looking decorations with a curated overprint geometry background system.
5. Use translucent foreground surfaces so the decorative geometry can show through without harming readability.
6. Use richer mid-century palette accents for group members, reactions, poster fallbacks, and decorative motifs.

The reference image is approved as the visual direction for this supplement. Codex should use the image for visual intent, but must implement from this written specification.

---

# 1. Core UX correction

## 1.1 Problem

The current prototype behaves too much like a mobile webpage:

- key screens stack content vertically
- completed steps remain visible
- users must scroll past old information to reach the active step
- the keyboard hides movie results
- user-facing screens include development/milestone language
- decorative background treatment is not yet consistent or premium

The target experience should feel like a mobile app.

## 1.2 New design rule

```text
Screen → decision → next screen
```

Not:

```text
Long page → scroll → completed section → scroll → next section
```

The app should use a fixed-height mobile app shell with focused panels. Full-page scrolling should be avoided in core flows.

# 2. App-like layout architecture

## 2.1 Core shell

Create reusable app-shell primitives.

Required components:

```text
AppShell
WizardShell
FixedHeader
FixedFooterAction
ScrollRegion
CompactSummaryCard
OverprintBackground
TranslucentSurface
```

## 2.2 Mobile viewport contract

Design primarily for:

```text
Width: 360px–430px
Height: 700px–950px
```

Use dynamic viewport height:

```css
.app-shell {
  width: 100vw;
  height: 100dvh;
  min-height: 100dvh;
  overflow: hidden;
  display: flex;
  flex-direction: column;
  background: var(--color-bg-page);
  position: relative;
  isolation: isolate;
}
```

Do not rely only on `100vh`, because mobile browser chrome and keyboards can make it unreliable.

## 2.3 Header

```text
Height: 52px–56px
Width: 100%
Flex: 0 0 auto
Border-bottom: 1px solid --color-border-subtle
Background: translucent paper surface, usually 94–98% opaque
Padding: 0 16px
Display: flex
Align-items: center
Justify-content: space-between
Z-index: 20
```

Header content:

- left: back button, menu button, or brand mark
- centre: group/page title
- right: participant badge, close button, or menu action

Keep the header visually light.

## 2.4 Main app region

```css
.app-main {
  flex: 1 1 auto;
  min-height: 0;
  overflow: hidden;
  display: flex;
  flex-direction: column;
  position: relative;
  z-index: 1;
}
```

Important implementation rule:

```text
Always use min-height: 0 on flex children that contain scroll regions.
```

## 2.5 Fixed footer action

For wizard flows and forms:

```css
.fixed-footer-action {
  flex: 0 0 auto;
  border-top: 1px solid var(--color-border-subtle);
  background: rgba(251, 247, 239, 0.96);
  padding: 12px 16px calc(12px + env(safe-area-inset-bottom));
  position: relative;
  z-index: 20;
}
```

Buttons and text inside the footer must remain fully opaque.

# 3. Scroll rules

## 3.1 Default

Avoid full-page body scrolling in core app flows.

Use:

```text
Fixed header
Flexible active content area
Fixed footer action
Internal scroll region only where required
```

## 3.2 Scrolling is allowed only in specific regions

Allowed internal scroll regions:

1. Movie search results.
2. Recommendation feed.
3. Long participant lists.
4. Movie detail content if metadata exceeds available height.
5. Admin invite list.
6. Expanded reason list or note editor if necessary.

Do not allow completed wizard steps to create long stacked pages.

## 3.3 Scroll region CSS

```css
.scroll-region {
  flex: 1 1 auto;
  min-height: 0;
  overflow-y: auto;
  overscroll-behavior: contain;
  -webkit-overflow-scrolling: touch;
}
```

# 4. Wizard architecture for Add Recommendation

The add recommendation flow must be a true wizard, not a long stacked form.

## 4.1 Exact five-step model

Use exactly five steps:

```text
Step 1: Search and select movie
Step 2: Choose target
Step 3: Choose reason and optional note
Step 4: Review recommendation
Step 5: Saved confirmation
```

The optional note belongs to Step 3.

Do not display six visible steps.

## 4.2 Wizard shell layout

```text
[FixedHeader: 52–56px]
[CompactProgress: 32–36px]
[ActiveStepBody: flex, min-height 0, internal scroll if needed]
[FixedFooterAction: 64–76px + safe area, where needed]
```

## 4.3 Progress indicator

```text
Height: 32px–36px
Active step: --color-accent
Completed steps: --color-accent-soft or charcoal
Inactive steps: muted neutral
```

Do not make the progress indicator visually heavier than the current task.

## 4.4 Completed steps collapse

Completed steps must render as compact summaries only.

Never keep a completed full form visible above the current step.

Use a compact summary card:

```text
Height: 76px–96px
Padding: 10px–12px
Border: 1px solid --color-border-subtle
Background: translucent paper surface, 92–96% opaque
Grid: 52px minmax(0, 1fr) auto
Gap: 10px
```

# 5. Step 1 — Search and select movie

## 5.1 Goal

The user should be able to type, see results, and select a movie without closing the keyboard or scrolling the whole page.

## 5.2 Layout

```text
Header: fixed
Progress: compact
Search block: compact
Results list: flexible internal scroll
Footer: absent until movie selected
```

## 5.3 Focused input behaviour

When search input is not focused:

```text
Title: Choose a movie
Helper: Search for the film you want to recommend.
Search input
```

When input is focused:

```text
Collapse title/helper into compact mode.
Show only:
- compact label: Search movies
- search input
- results immediately below
```

## 5.4 Remove large Search button

Do not use a large separate `Search` button on mobile.

Use search-as-you-type:

```text
Debounce: 300ms
Start search after: 2+ characters
Enter key: triggers search immediately
Loading: show skeleton rows directly below input
```

## 5.5 Results region

Results must start immediately below the search input.

```text
Result list:
- flex: 1
- min-height: 0
- overflow-y: auto
- padding-top: 8px
- padding-bottom: 16px
```

Result card:

```text
Grid: 64px poster + flexible text
Padding: 10px–12px
Gap: 12px
Border: 1px solid --color-border-subtle
Background: translucent paper surface, 94–97% opaque
Overview: max 2 lines
No TMDB ID in normal user-facing UI
```

## 5.6 Keyboard acceptance criteria

On a 390px-wide phone with keyboard open:

- search input remains visible
- at least the first two results are visible
- the user can select a result without closing the keyboard
- no full-page scrolling is required

# 6. After movie selection

## 6.1 Required behaviour

When the user selects a movie:

1. Save selected movie in flow state.
2. Clear or hide search results.
3. Collapse Step 1 into a compact selected-movie summary.
4. Move to Step 2.
5. Ensure Step 2 is immediately visible.

Do not leave the search form and search result list visible.

## 6.2 Compact selected movie summary

```text
SelectedMovieSummary:
- Height: 76px–96px
- Width: 100%
- Padding: 10px–12px
- Border: 1px solid --color-border-subtle
- Background: translucent paper surface, 94–96% opaque
- Border radius: 4px
- Grid columns: 52px minmax(0, 1fr) auto
- Gap: 10px
```

Poster:

```text
Width: 52px
Aspect ratio: 2 / 3
Object-fit: cover
```

Text:

```text
Label: Selected movie
Title: IBM Plex Sans Condensed, 18px–20px, uppercase, max 2 lines
Metadata: year · genres, 12px–13px, muted
```

Action:

```text
Change
```

The `Change` action returns to Step 1 and restores the previous query where practical.

# 8. Step 3 — Choose reason and optional note

## 8.1 Layout

```text
Top: compact selected movie summary, smaller if necessary
Main: reason chips
Optional note: collapsed by default or compact textarea
Footer: Continue
```

## 8.2 Reason chips

Use wrapped pill chips, not square icon tiles.


- keep selected movie/reason context compact

- keep Continue reachable where practical

# 13. Curated overprint geometry background system

## 13.1 Decision

Use a curated library of standalone overprint SVG decorations, randomly selected per page/session.

Do not procedurally generate arbitrary new decorations.

Do not use AI-generated random backgrounds at runtime.

Use motifs based on the supplied approved design-sheet image.

## 13.2 Why

Autogenerated decorations have been disappointing. The design needs controlled quality.

The correct model:

```text
Curated SVG motif library + seeded selection + responsive placement
```

Not:

```text
Procedural random art generation
```

## 13.3 Motif library

Create 12–24 curated inline SVG motifs inspired by the reference sheet.

Initial MVP set:

```text
1. bottom-left overlapping orbs
2. bottom-right overlapping orbs
3. top-right vertical rules + circle
4. large cropped arch
5. keyhole stack
6. triangle peaks
7. soft semicircle landscape
8. corner crop blocks
9. vertical stripe cluster
10. poster fallback motif
11. empty-state motif
12. brand mark motif
```

Additional later variants:

```text
circles-orbs-01
circles-orbs-02
arches-01
arches-02
keyhole-01
keyhole-02
peaks-01
stacks-01
verticals-01
corner-crop-01
corner-crop-02
banner-01
```

## 13.4 Approved palette

Motifs may use:

```text
rose
teal
green
orange
purple
olive
charcoal
paper
```

Token values:

```css
:root {
  --color-accent: #C95F82;
  --color-accent-teal: #5F93AA;
  --color-accent-green: #5B9277;
  --color-accent-orange: #C8793A;
  --color-accent-purple: #6B4568;
  --color-accent-olive: #8A8A54;
  --color-text-primary: #2D2D27;
  --color-bg-page: #F1EADF;
}
```

## 13.5 Motif visual style

Signature graphic style:

```text
Restrained mid-century overprint geometry.
Use overlapping circles, arches, triangles, keyholes, semicircles, corner crops, and vertical rules with a subtle screen-printed / risograph-like texture.
Shapes should have slight ink variation, faint paper grain, and gently imperfect hand-cut edges.
Colours may overlap with translucent multiply-like interaction.
Keep the effect warm and tactile, not distressed, dirty, or grungy.
```

## 13.6 Implementation

Use inline SVG React components.

Recommended component:

```tsx
<OverprintBackground
  route="group-home"
  seed={`${groupId}:${participantId}`}
  density="medium"
/>
```

Recommended props:

```ts
type OverprintBackgroundProps = {
  route: string;
  seed: string;
  density?: "none" | "subtle" | "medium" | "bold";
  className?: string;
};
```

Motif component:

```ts
type OverprintMotifProps = {
  variant:
    | "brandMark"
    | "cornerOrbs"
    | "bottomOrbs"
    | "largeArch"
    | "keyholeStack"
    | "trianglePeaks"
    | "semicircleLandscape"
    | "verticalRules"
    | "posterFallback"
    | "emptyState";
  size?: "sm" | "md" | "lg";
  intensity?: "subtle" | "standard" | "bold";
  className?: string;
};
```

## 13.7 Background layer structure

```css
.app-shell {
  position: relative;
  isolation: isolate;
  overflow: hidden;
  background: var(--color-bg-page);
}

.background-motif-layer {
  position: absolute;
  inset: 0;
  z-index: 0;
  pointer-events: none;
  overflow: hidden;
}

.app-content {
  position: relative;
  z-index: 1;
}
```

Motifs must be:

```text
aria-hidden="true"
pointer-events: none
decorative only
never required to understand UI
```

## 13.8 Blend and texture

Shape opacity:

```text
0.62–0.85
```

Use `mix-blend-mode: multiply` only inside decorative motif layers and test carefully.

Use subtle grain only if performance and appearance remain good:

```text
grain opacity: 2–4%
```

Do not let blend modes affect text, cards, or controls.

Use `isolation: isolate`.

# 14. Responsive motif behaviour

Motifs must not be designed as one full-screen image.

They should be responsive edge/corner compositions.

## 14.1 Mobile

```text
Motif size: 160–280px
Placement: bottom-left, bottom-right, top-right crop, lower third
Visibility: subtle to medium
```

## 14.2 Tablet

```text
Motif size: 240–380px
Placement: corners and edges
```

## 14.3 Desktop

```text
Motif size: 360–620px
Placement: wider corners, side bands, bottom landscapes
```

## 14.4 Safe placements

Good:

```text
bottom-right
bottom-left
top-right, partially cropped
lower third behind empty state
card corner accent
poster fallback centre
```

Avoid:

```text
directly behind paragraph text
behind input fields
behind primary CTA
behind dense search results
behind validation/error messages
```

# 15. Seeded randomness

## 15.1 Goal

Backgrounds should feel alive and varied without becoming chaotic or changing unexpectedly.

Use stable seeded randomness.

Recommended seed:

```ts
const seed = `${route}:${groupId ?? "public"}:${participantId ?? "anon"}`;
```

Use the seed to choose:

```text
motif variant
placement
scale
rotation
colourway
opacity
```

Do not randomise:

```text
core layout
foreground opacity
text contrast
button placement
scroll behaviour
```

## 15.2 Stability rule

A motif must not change on every React render.

It may change when:

- route changes
- group changes
- participant changes
- new session begins, if intentionally configured

## 15.3 Variation limits

```text
Rotation: -4deg to +4deg
Scale: 0.8–1.4
Shape opacity: 0.35–0.75 at overall motif level
Individual shape opacity: 0.62–0.85
```

# 16. Translucent foreground surfaces

## 16.1 Principle

Let the backgrounds show through, but keep content readable.

Do not apply `opacity` to entire cards or panels.

Bad:

```css
.card {
  opacity: 0.9;
}
```

This makes text and controls transparent.

Good:

```css
.card {
  background: rgba(251, 247, 239, 0.94);
}
```

Text and controls remain fully opaque.

## 16.2 Surface tokens

Add:

```css
:root {
  --surface-translucent: rgba(251, 247, 239, 0.92);
  --surface-translucent-strong: rgba(251, 247, 239, 0.96);
  --surface-translucent-soft: rgba(251, 247, 239, 0.86);
}
```

## 16.3 Usage

```text
Primary content cards: --surface-translucent-strong
Search/results/forms: --surface-translucent-strong or opaque
Secondary panels: --surface-translucent
Empty states: --surface-translucent-soft
Headers/footers: 94–98% opaque
Buttons: 100% opaque
Inputs: 94–98% opaque
```

Suggested opacity levels:

```text
Primary content cards: 94–97%
Search/results/forms: 96–98%
Empty states: 86–92%
Decorative panels: 84–90%
Buttons and text: 100%
```

## 16.4 Acceptance criteria

- Background motifs are visible enough to add warmth and identity.
- Text remains readable.
- UI does not feel busy.
- Motifs never compete with the active task.
- Page loads do not visibly jump.

# 17. Richer mid-century colour usage

## 17.1 Principle

Use rose as the main accent, but bring more mid-century colour into small repeatable elements.

Do not add more large coloured surfaces.

## 17.2 Group member colours

Use colour liberally for group members.

Each participant gets a deterministic colour from the palette.

```css
:root {
  --avatar-rose: #C95F82;
  --avatar-teal: #5F93AA;
  --avatar-green: #5B9277;
  --avatar-orange: #C8793A;
  --avatar-purple: #6B4568;
  --avatar-olive: #8A8A54;
}
```

Avatar rules:

```text
Colour determined by avatarSeed or participantId
Stable across sessions
Used in avatar badge, participant chips, recommender markers, and group member list
Text/icon contrast must remain accessible
```

## 17.3 Reason chip tints

Selected chip remains rose.

Unselected/category chips may use very soft tint backgrounds:

```css
:root {
  --chip-soft-rose: #F0D4DD;
  --chip-soft-teal: #D8E6EA;
  --chip-soft-green: #DCE8DF;
  --chip-soft-orange: #EEDCC7;
  --chip-soft-purple: #E2D4E0;
  --chip-soft-olive: #E3E4CF;
}
```

Use sparingly. Avoid turning the whole chip field into a rainbow.

## 17.4 Reaction colours

Use subtle semantic colours for reactions.

```css
:root {
  --reaction-want: #5F93AA;
  --reaction-maybe: #C8793A;
  --reaction-seen: #6F665B;
  --reaction-loved: #C95F82;
  --reaction-not-for-me: #6B4568;
}
```

Use these as:

- icon colour
- selected chip border/fill
- small count markers

Do not use them as large card backgrounds.

## 17.5 Section divider colour bars

Add small coloured bar clusters beside section headings.

Example:

```text
RECENT RECOMMENDATIONS    [rose] [teal] [orange]
```

Rules:

```text
Height: 3px
Width: 18px each
Gap: 4px
Opacity: 85%
Use 2–4 bars max
```

This is one of the safest ways to add mid-century character.

# 18. Poster fallback artwork

## 18.1 Problem

Plain poster placeholders look unfinished.

## 18.2 Requirement

Missing poster fallback must use a designed overprint motif.

Fallback should feel like a small abstract paperback/poster, not a blank placeholder.

Use:

```text
paper background
charcoal centre mark
rose + teal + olive/orange accent shapes
subtle overprint texture
```

Poster fallback dimensions must match the poster component.

```text
Aspect ratio: 2 / 3
Object-fit equivalent
Border: 1px solid --color-border-subtle
```

# 19. Visual QA checklist

## 19.1 App-like flow

- [ ] Core wizard flows use `100dvh` shell.
- [ ] Body does not become a long stacked document.
- [ ] Only active step is expanded.
- [ ] Completed steps collapse into compact summaries.
- [ ] Search results are visible above keyboard.
- [ ] Selecting a movie advances to Step 2.
- [ ] Search form/results disappear after selection.
- [ ] Primary actions remain reachable.

## 19.2 Background motifs

- [ ] Motifs come from curated SVG library.
- [ ] Motifs do not change on every render.
- [ ] Motifs are responsive and crop safely.
- [ ] Motifs do not sit behind critical text or inputs.
- [ ] Background layer is non-interactive and `aria-hidden`.
- [ ] Blend modes do not affect foreground content.

## 19.3 Translucent surfaces

- [ ] Cards use semi-transparent backgrounds, not `opacity`.
- [ ] Text and controls remain fully opaque.
- [ ] Dense forms/results remain highly legible.
- [ ] Empty states allow more motif visibility.
- [ ] Contrast remains acceptable.

## 19.4 Colour

- [ ] Participant avatars use deterministic palette colours.
- [ ] Rose remains the primary CTA accent.
- [ ] Secondary colours appear in avatars, motifs, reaction accents, poster fallbacks, and small dividers.
- [ ] No large rainbow surfaces.
- [ ] UI remains calm and premium.

# 21. Anti-patterns to avoid

Do not:

- procedurally generate arbitrary decorations
- use static full-screen background images
- allow motif changes on every render
- put decoration behind active inputs
- use opacity on whole cards
- make text or controls translucent
- stack every wizard step on a long page
- leave completed search UI visible after selection
- require users to close keyboard to see results
- let background motifs compete with movie posters or text
- use large multi-colour panels that make the app noisy
- expose development/milestone language in the UI

---

# 22. Final design intent

The app should feel like:

```text
A calm, app-like private film recommendation tool with warm paper surfaces, translucent editorial panels, curated overprint geometry, colourful trusted-circle identities, and focused wizard interactions.
```

It should not feel like:

```text
A long mobile web page with decorative assets added behind it.
```

The supplied reference image is the approved visual direction for:

- curated overprint motif backgrounds
- translucent foreground surfaces
- richer mid-century colour accents
- app-like screen composition
- reduced scrolling
- warm, human, trustworthy tone
